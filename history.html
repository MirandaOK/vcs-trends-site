VCS evolution

  1. 1972: Development of the Source Code Control System (SCCS)
  2. 1982: Initial release of the Revision Control System (RCS)
  3. 1986: Initial release of the Concurrent Versions System (CVS)
  4. 2000: Initial release of the Subversion (SVN)
  5. 2005: Initial release of the Git and Mercurial
  6. 2008: Launch of the GitHub
  7. 2010s: Wide adoption of Git and migration from older systems
  8. Late 2010s: GitOps and modern practices
  9. 2020s: AI and large-scale collaboration

Version control systems can be centralized or distributed:

  • Centralized Version Control (CVCS) - the repository is located on a single, master server. Commits require connection to the master server. Centralized tools are for example RCS, CVS and Subversion.

  • Distributed Version Control (DVCS) - devs have their own repositories. Commits can be done offline and later pushed to share with others. Distributed tools are for example Git or Mercurial.

Distributed VCS was a game-changer because it allowed developers to work asynchronously and allow collaboration at a very huge scale (eg. Git).

Use of the tools:

  • RCS (Revision Control System) - mostly used for simple, small projects, it's not meant for collaborative network use.
  • CVS (Concurrent Versions System) - small to medium team projects, currently, not used much by teams due to several problems like weak branching/merging support.
  • Subversion (SVN) - was very popular in 2006, however it was difficult to use for very large teams.
  • Git - any scale projects, today it's very popular thanks to its great performance and flexibility, it works very well for source projects and DevOps workflows and provides extremely fast branching and merging.
  • Mercurial - any scale projects, although it's efficient and user-friendly it didn't get as popular as Git, partially because initially GitHub didn't supported it.
